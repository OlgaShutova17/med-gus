## Ключевая идея продукта

Большинство медицинских приложений показывают **данные**.
Этот продукт показывает **расследование**: почему пользователю плохо и что с этим делать.

Центральная метрика успеха:

> Пользователь получает ответ на вопрос **«Почему мне плохо?»**

---

## Философия UX

Продукт построен вокруг **проблемы**, а не вокруг данных:

```
Обычные health-apps:   Анализ → Анализ → Анализ
Наш продукт:           Проблема → Гипотезы → Расследование → Вывод
```

Это принципиальное отличие: пользователь видит не набор файлов и цифр, а **ход лечения** конкретной проблемы со своей историей, версиями причин и их проверкой.

---

## Модель данных с точки зрения UX

```
Пользователь
   └── Проблема (Тематика)
         ├── Симптомы
         ├── Гипотезы (предполагаемые причины)
         │     └── Курсы лечения
         └── События (хронология)
               ├── Консультация специалиста
               ├── Анализ (с показателями)
               ├── Исследование (МРТ, КТ, УЗИ)
               └── Другое
                     └── Резолюция («Помогло» / «Не помогло»)
```

## Дуальная модель интерфейса (ключевое решение)

Проблема классического timeline: пользователь видит события, но не понимает их смысл.

Решение — **два взаимодополняющих представления**:

### Timeline — «что происходило»
```
1 марта   — начало симптомов (усталость, головокружение)
10 марта  — анализ крови → ферритин 10 мкг/л (ниже нормы)
15 марта  — консультация терапевта → направление на дообследование
20 марта  — начало курса лечения (препараты железа)
```

### Гипотезы — «почему это происходит»
```
Дефицит железа       вероятность 72% ← подтверждают: ферритин↓, гемоглобин↓
Гипотиреоз           вероятность 15% ← опровергает: ТТГ в норме
Хронический стресс   вероятность 13%
```

Каждое событие в Timeline **влияет на гипотезы**: подтверждает, опровергает или нейтрально.

## Ключевые UX-функции

### 1. Health Timeline
Единая хронология: симптом → анализ → диагноз → лечение → результат.
Пользователь видит полный контекст, а не разрозненные файлы.

### 2. AI Second Opinion
AI анализирует всю историю и формирует «второе мнение»:
- Вероятные причины (с объяснением)
- Следующий рекомендуемый шаг
- К какому специалисту обратиться

### 3. Medical Pattern Detection
AI находит паттерны в данных:
```
Пример: мигрень регулярно возникает через 2 дня после плохого сна
```

### 4. Progress of Investigation
Прогресс-бар диагностики — насколько подтверждена ведущая гипотеза:
```
Диагностика проблемы «Хроническая усталость»
Дефицит железа — 72% уверенности [███████░░░]
```

### 5. Evidence Chips
Визуальные подтверждения/опровержения прямо на карточке гипотезы:
```
✓ ферритин ↓   ✓ гемоглобин ↓   ✓ усталость   ✗ ТТГ норма
```

## Пользовательский путь (User Journey)

```
Регистрация
   ↓
Создание проблемы (например, «Хроническая усталость»)
   ↓
Добавление симптомов
   ↓
Добавление гипотезы (например, «Дефицит железа»)
   ↓
Загрузка анализа (файл PDF или ручной ввод)
   ↓
Получение AI-интерпретации
   ↓
Добавление консультации специалиста
   ↓
Создание курса лечения
   ↓
Отметка резолюции («Помогло» / «Не помогло»)
   ↓
AI обновляет вероятности гипотез
```

---

## Принципы интерфейса

1. **Минималистичный UI** — интерфейс не требует обучения; пользователь должен разобраться интуитивно
2. **Проблема-ориентированность** — всё группируется вокруг конкретной проблемы, а не по типу данных
3. **AI как помощник, не замена врача** — все AI-рекомендации сопровождаются дисклеймером
4. **Подтверждение распознанных данных** — после OCR пользователь обязательно верифицирует извлечённые значения
5. **Доступность** — ≤1 секунды на загрузку истории и аналитики

---

## Нефункциональные требования UX

- Конфиденциальность медицинских данных (ФЗ о персональных данных)
- Изоляция данных между пользователями (row-level security)
- Механизм согласия на обработку данных при регистрации
